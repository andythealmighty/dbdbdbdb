데이터베이스란 무엇인가요? 🤔

데이터베이스는 여러분의 정보를 저장하는 디지털 창고예요. 우리 집에 있는 서랍장처럼, 각 서랍마다 특정 물건들을 정리해서 보관하는 것과 비슷합니다.

📱 여러분이 스마트폰에서 연락처를 저장할 때, 그 연락처는 사실 작은 데이터베이스에 저장되고 있어요!

🛒 온라인 쇼핑몰에서 물건을 살 때, 여러분의 주문 정보는 데이터베이스에 저장됩니다.

🏦 은행 계좌의 잔액, 입출금 내역도 모두 데이터베이스에 저장되어 있죠!

쉽게 생각하기

스마트폰 연락처부 = 작은 데이터베이스

엑셀 spreadsheet = 간단한 데이터베이스

도서관 책 관리 시스템 = 큰 데이터베이스

DB 구조 쉽게 이해하기 📋

데이터베이스는 여러 테이블로 구성되어 있어요. 테이블은 엑셀 시트처럼 행(가로줄)과 열(세로줄)로 구성됩니다.

📋 테이블 = 정보를 담는 표 (예: 사용자 테이블, 매물 테이블)

📝 열(Column) = 정보의 종류 (예: 이름, 이메일, 전화번호)

📌 행(Row) = 하나의 데이터 세트 (예: 한 명의 사용자 정보)

🔑 기본 키(Primary Key) = 각 행을 구분하는 고유 식별자 (예: 사용자 ID)

🔗 외래 키(Foreign Key) = 다른 테이블과 연결하는 키 (예: 사용자 ID 참조)

초간단 예시: 연락처 앱
id (기본 키) 이름 전화번호 그룹
1 김철수 010-1234-5678 친구
2 이영희 010-8765-4321 가족

사용자 사용자 정보 (Users) 👤

사용자 관련 테이블들은 시스템을 사용하는 모든 사람들의 정보를 저장해요. 프랜차이즈 담당자, 부동산 중개인, 점주 등 각 사용자의 기본 정보와 역할별 세부 정보를 관리합니다.
users 테이블 (모든 사용자의 기본 정보) ▼
모든 사용자의 공통된 기본 정보를 저장하는 테이블입니다.
id (INT)
고유 사용자 ID - 각 사용자를 구분하는 고유 번호예요
email (VARCHAR(255))
로그인 이메일 - 로그인에 사용하는 이메일 주소예요 (중복 불가)
password_hash (VARCHAR(255))
암호화된 비밀번호 - 보안을 위해 실제 비밀번호가 아닌 암호화된 값을 저장해요
name (VARCHAR(100))
사용자 이름 - 실명 또는 닉네임이에요
phone (VARCHAR(20))
연락처 - 전화번호예요
role (ENUM)
사용자 역할 - 프랜차이즈, 부동산, 예비점주, 관리자 중 하나예요
is_deleted (BOOLEAN)
삭제 여부 - 계정 삭제 여부를 표시해요
created_at (DATETIME)
생성일시 - 계정이 만들어진 날짜와 시간이에요

실제 예시: 김철수(id: 1)님은 부동산 중개사(role)로 가입했고, 이메일은 kim@example.com, 전화번호는 010-1234-5678입니다.

franchise_profiles 테이블 (프랜차이즈 담당자 프로필) ▼
프랜차이즈 담당자의 추가 정보를 저장하는 테이블입니다.
user_id (INT)
users.id 참조 - users 테이블의 어떤 사용자인지 연결해요
company_name (VARCHAR(255))
프랜차이즈 회사 이름 - 담당하는 프랜차이즈 기업명이에요
position (VARCHAR(100))
직위 - 회사 내 직책이에요
bio (TEXT)
자기소개 - 자신에 대한 소개글이에요

실제 예시: 박지민(user_id: 3)님은 '맛있는 치킨' 프랜차이즈의 '출점 담당 매니저'이며, 자기소개에 "10년 경력의 출점 전문가입니다"라고 적었습니다.

agent_profiles 테이블 (부동산 중개인 프로필) ▼
부동산 중개인의 추가 정보를 저장하는 테이블입니다.
user_id (INT)
users.id 참조 - users 테이블의 어떤 사용자인지 연결해요
agency_name (VARCHAR(255))
부동산 회사 이름 - 소속된 부동산 중개사무소 이름이에요
license_number (VARCHAR(100))
공인중개사 등록번호 - 공인중개사 자격을 증명하는 번호예요
bio (TEXT)
자기소개 - 자신에 대한 소개글이에요

실제 예시: 김철수(user_id: 1)님은 '행복한 부동산'의 공인중개사로, 면허번호는 '서울12345'이며, "상가 전문 중개사입니다"라고 소개했습니다.

owner_profiles 테이블 (예비 점주 프로필) ▼
매장을 열고자 하는 예비 점주의 추가 정보를 저장하는 테이블입니다.
user_id (INT)
users.id 참조 - users 테이블의 어떤 사용자인지 연결해요
interests (TEXT)
관심 프랜차이즈/업종/지역 등 - 어떤 종류의 매장에 관심이 있는지 저장해요

실제 예시: 이영희(user_id: 2)님은 "카페, 베이커리 / 강남, 송파구 / 5천만원~1억 예산"이라고 자신의 관심사를 등록했습니다.

쉽게 이해하기

🔑 users 테이블은 모든 사용자의 공통 정보를 저장해요.

👩‍💼 franchise_profiles는 프랜차이즈 담당자의 추가 정보를 저장해요.

🏢 agent_profiles는 부동산 중개인의 추가 정보를 저장해요.

💰 owner_profiles는 예비 점주의 추가 정보를 저장해요.

📌 모든 프로필 테이블은 user_id로 사용자 테이블과 연결됩니다!

매물 매물 정보 (Properties) 🏢

매물 관련 테이블들은 판매 또는 임대 중인 부동산 정보와 프랜차이즈에서 요청한 매물 조건을 저장해요. 위치, 크기, 가격 등 상세 정보를 관리합니다.
properties 테이블 (매물 정보) ▼
판매 또는 임대 중인 부동산 매물 정보를 저장하는 테이블입니다.
id (INT)
매물 ID - 각 매물을 구분하는 고유 번호예요
title (VARCHAR(255))
매물 제목 - 매물을 소개하는 제목이에요
address (VARCHAR(500))
상세 주소 - 매물의 정확한 위치예요
region_code (VARCHAR(10))
행정동 코드 - 행정구역 코드로, 지역을 구분하는 번호예요
area_m2 (FLOAT)
면적 (㎡) - 매물의 크기를 제곱미터로 표시해요
area_pyeong (FLOAT)
면적 (평) - 매물의 크기를 평으로 표시해요
deposit (INT)
보증금 - 임대 계약 시 필요한 보증금이에요
monthly_rent (INT)
월세 - 매달 내야 하는 임대료예요
sale_price (INT)
매매가 - 매물을 구입하는 가격이에요
floor (INT)
층수 - 매물이 위치한 층이에요
building_type (VARCHAR(100))
건물 유형 - 상가, 사무실 등 건물의 종류예요
status (ENUM)
상태 - 현재 매물의 상태로, '판매중', '계약중', '판매완료' 등이 있어요
agent_id (INT)
등록한 부동산 사용자 ID - 이 매물을 등록한 부동산 중개인을 가리켜요

실제 예시: '강남역 1분 거리 신축 상가'(id: 5)는 서울시 강남구에 위치한 1층 상가로, 면적은 30㎡(약 9평), 보증금 5천만원에 월세 200만원이며 김철수 중개사(agent_id: 1)가 등록했습니다.

property_requests 테이블 (매물 요청) ▼
프랜차이즈에서 특정 조건의 매물을 찾아달라고 요청하는 정보를 저장하는 테이블입니다.
id (INT)
요청 ID - 각 요청을 구분하는 고유 번호예요
franchise_id (INT)
요청한 프랜차이즈 사용자 ID - 이 요청을 한 프랜차이즈 담당자를 가리켜요
title (VARCHAR(255))
요청 제목 - 매물 요청의 제목이에요
region_preference (VARCHAR(255))
선호 지역 - 매물을 찾고 싶은 지역이에요
status (ENUM)
상태 - 요청의 현재 진행 상태예요 (대기중, 진행중, 완료됨, 취소됨)
due_date (DATE)
마감일 - 매물 찾기를 완료해야 하는 날짜예요

실제 예시: '맛있는 치킨' 프랜차이즈(franchise_id: 3)는 '강남 지역 신규 매장 물건 요청'이라는 제목으로, 강남역 반경 1km 이내에서 신규 매장 물건을 찾아달라고 요청했습니다.

request_preferences 테이블 (요청 상세 조건) ▼
매물 요청의 상세 조건을 저장하는 테이블입니다.
id (INT)
ID - 각 조건 세트를 구분하는 고유 번호예요
request_id (INT)
property_requests.id 참조 - 어떤 요청에 대한 조건인지 연결해요
property_type (VARCHAR(100))
매물 유형 - 원하는 매물의 종류예요 (상가, 사무실 등)
min_area, max_area (FLOAT)
최소/최대 면적 - 원하는 매물 크기의 범위예요
min_budget, max_budget (INT)
예산 범위 - 사용 가능한 금액의 범위예요

실제 예시: '맛있는 치킨'의 요청(request_id: 5)은 1층 상가, 최소 20평 이상, 월세 최대 300만원, 보증금 최대 1억원의 조건을 가지고 있습니다.

쉽게 이해하기

🏢 properties 테이블은 판매/임대 중인 부동산의 모든 정보를 담아요.

🔍 property_requests는 프랜차이즈가 찾고 있는 매물의 조건을 담아요.

📋 request_preferences는 매물 요청의 상세 조건을 더 자세히、담아요.

이렇게 생각하세요: "강남에 치킨집 낼 매장 좀 찾아주세요"(property_requests)와 "1층, 30평, 보증금 5천 이하로요"(request_preferences)가 함께 있는 거죠!

AI 분석 AI 분석 보고서 (AI Reports) 🤖

AI 관련 테이블들은 인공지능이 매물을 분석한 결과와 점수를 저장해요. 입지, 상권, 경쟁, 수익성 등 다양한 측면에서 평가합니다.
ai_reports 테이블 (AI 분석 보고서 기본 정보) ▼
AI 분석 보고서의 기본 정보를 저장하는 테이블입니다.
id (INT)
보고서 ID - 각 보고서를 구분하는 고유 번호예요
property_id (INT)
분석 대상 매물 ID - 어떤 매물을 분석했는지 연결해요
request_id (INT)
분석 대상 요청 ID - 어떤 요청을 분석했는지 연결해요
report_type (ENUM)
분석 대상 구분 - 매물 분석인지 요청 분석인지 구분해요
overall_score (DECIMAL(5,2))
종합 점수 - AI가 매긴 전체 평가 점수예요
summary (TEXT)
요약 설명 - 분석 결과를 간략히 설명한 내용이에요

실제 예시: 강남역 상가(property_id: 5)에 대한 AI 분석 보고서(id: 10)는 종합 점수 85점으로, "유동인구가 많고 상권이 활성화된 우수 입지"라고 요약되었습니다.

ai_report_location 테이블 (입지 분석) ▼
매물 위치에 대한 AI의 입지 분석 결과를 저장하는 테이블입니다.
report_id (INT)
ai_reports.id 참조 - 어떤 보고서에 대한 분석인지 연결해요
location_score (DECIMAL(5,2))
입지 점수 - 위치에 대한 평가 점수예요
population_analysis (JSON)
시간대별 유동인구 - 주변의 시간별 유동인구 데이터예요
demographics (JSON)
인구 통계 - 주변 주민의 연령, 성별, 소득 데이터예요
strengths, weaknesses (TEXT)
장점, 단점 - 입지의 강점과 약점 분석이에요

실제 예시: 강남역 상가에 대한 입지 분석은 입지 점수 90점으로, "지하철역과 가까워 접근성 우수"라는 장점과 "주변 임대료가 높음"이라는 단점을 가지고 있습니다.

ai_report_profit 테이블 (수익성 분석) ▼
매물의 예상 수익성에 대한 AI의 분석 결과를 저장하는 테이블입니다.
report_id (INT)
ai_reports.id 참조 - 어떤 보고서에 대한 분석인지 연결해요
profit_score (DECIMAL(5,2))
수익성 점수 - 예상 수익에 대한 평가 점수예요
expected_revenue_min/avg/max (INT)
예상 매출 - 최소/평균/최대 예상 매출액이에요
break_even_point (INT)
손익분기점 - 투자금이 회수되는 예상 시점이에요

실제 예시: 강남역 상가에 치킨집을 열 경우, 수익성 점수 75점, 월 평균 매출 2천만원, 손익분기점 18개월로 예상됩니다.

쉽게 이해하기

📊 ai_reports는 AI 분석의 기본 정보와 종합 점수를 담아요.

📍 ai_report_location은 위치와 입지를 분석해요 (사람들이 얼마나 많이 지나가는지, 어떤 사람들이 사는지).

💰 ai_report_profit은 얼마나 돈을 벌 수 있을지를 예측해요.

생각해보세요: AI가 "이 위치에 치킨집을 열면 월 2천만원 매출이 예상되고, 1년 6개월이면 본전을 뽑을 수 있어요"라고 알려주는 거예요!

계약 계약 정보 (Contracts) 📝

계약 관련 테이블들은 부동산 거래 계약에 관한 모든 정보를 저장해요. 계약 당사자, 조건, 계약서 등을 관리합니다.
contracts 테이블 (계약 기본 정보) ▼
계약의 기본 정보를 저장하는 테이블입니다.
id (INT)
계약 ID - 각 계약을 구분하는 고유 번호예요
property_id (INT)
대상 매물 - 계약 대상인 매물을 연결해요
type (ENUM)
계약 유형 - 임대, 매매, 프랜차이즈 계약 중 하나예요
start_date, end_date (DATE)

실제 예시: 강남역 상가(property_id: 5)에 대한 임대 계약(id: 7)은 2023년 1월 1일부터 2025년 12월 31일까지, 보증금 5천만원, 월세 200만원의 조건을 가집니다.

실제 예시: 계약 7번에는 '맛있는 치킨' 프랜차이즈 담당자(franchise_id: 3), 김철수 부동산 중개사(agent_id: 1), 이영희 예비 점주(owner_id: 2)가 참여했습니다.

쉽게 이해하기

테이블 간의 관계 이해하기 🔄

데이터베이스에서 가장 중요한 것은 테이블들이 서로 어떻게 연결되어 있는지 이해하는 것이에요. 이것을 '관계'라고 합니다.

관계의 종류:

1. 일대일(1:1) 관계: 한 테이블의 하나의 행이 다른 테이블의 정확히 하나의 행과 연결됩니다.

예: 각 agent_profiles는 정확히 하나의 users 행과 연결됩니다.

2. 일대다(1:N) 관계: 한 테이블의 하나의 행이 다른 테이블의 여러 행과 연결됩니다.

예: 한 명의 부동산 중개인(users)은 여러 개의 매물(properties)을 등록할 수 있습니다.

3. 다대다(N:M) 관계: 두 테이블 사이에 중간 테이블을 두어 여러 행들이 서로 연결됩니다.

핵심 관계 이해하기

1. users ↔ profiles: 각 사용자는 하나의 프로필(프랜차이즈, 부동산, 점주)을 가집니다.

2. agents ↔ properties: 부동산 중개인은 여러 매물을 등록할 수 있습니다.

3. franchise ↔ property_requests: 프랜차이즈는 여러 매물 요청을 할 수 있습니다.

4. properties/requests ↔ ai_reports: 매물이나 요청에 대해 AI 분석 보고서가 생성됩니다.

5. properties ↔ contracts: 매물에 대해 계약이 체결됩니다.

6. users ↔ messages: 사용자들은 서로 메시지를 주고받습니다.

실생활 예시로 이해하기 📝

이제 실제 예시를 통해 이 모든 테이블이 어떻게 함께 작동하는지 살펴볼까요?
예시 1: 치킨 프랜차이즈 매장 찾기 ▼
'맛있는 치킨' 프랜차이즈가 새 매장을 찾는 과정을 데이터베이스 관점에서 살펴봅시다.
  1. '맛있는 치킨' 프랜차이즈 담당자 박지민씨는 users 테이블에 ID 3번으로 등록되어 있습니다.
  2. 그의 프로필 정보는 franchise_profiles 테이블에 저장되어 있어요.
  3. 박지민씨는 강남역 근처에 매장을 열고 싶어 property_requests 테이블에 새 요청을 등록했어요.
  4. 원하는 상세 조건(1층, 30평 이상 등)은 request_preferences 테이블에 저장되었어요.
  5. 요청을 본 부동산 중개인 김철수씨(agent_profiles)가 적합한 매물을 properties 테이블에서 찾아요.
  6. AI가 해당 매물을 분석하여 ai_reports와 관련 분석 테이블에 결과를 저장해요.
  7. 김철수씨는 박지민씨에게 chat_rooms과 messages를 통해 매물 정보를 전달해요.
  8. 박지민씨가 매물이 마음에 들면, contracts 테이블에 새 계약이 생성되고 contract_parties에 관련자들이 연결돼요.
  9. 모든 과정은 activity_logs에 기록됩니다.
예시 2: 예비 점주의 매장 찾기 ▼
카페를 열고 싶은 예비 점주가 적합한 매장과 프랜차이즈를 찾는 과정을 살펴봅시다.
  1. 이영희씨는 카페 창업을 희망하는 예비 점주로 users 테이블에 ID 2번으로 등록되어 있어요.
  2. 그녀의 관심사는 owner_profiles 테이블에 "카페, 강남, 5천만원 예산"으로 저장되어 있어요.
  3. 부동산 중개인 김철수씨는 properties 테이블에서 적합한 매물을 찾아 이영희씨에게 제안해요.
  4. 이영희씨는 해당 매물의 ai_reports를 통해 입지와 예상 수익성을 확인해요.
  5. 동시에 franchises 테이블에서 적합한 카페 프랜차이즈를 찾아봐요.
  6. 이영희씨, 김철수씨, 카페 프랜차이즈 담당자는 chat_rooms을 통해 소통해요.
  7. 모든 조건이 맞으면 contracts에 임대 계약이 생성되고, 이어서 프랜차이즈 계약도 생성돼요.
  8. 계약 관련 문서는 contract_documents에 저장돼요.
  9. 이영희씨는 notifications를 통해 진행 상황 알림을 받아요.
핵심 포인트

데이터베이스는 마치 퍼즐처럼 모든 정보가 서로 연결되어 있어요!

하나의 정보(예: 사용자)는 여러 다른 정보(매물, 메시지, 계약 등)와 연결됩니다.

이런 연결 관계 덕분에 "강남에 있는 김철수 중개사가 등록한 모든 매물"과 같은 복잡한 정보도 쉽게 찾을 수 있어요.

예시 3: 부동산 중개인의 매물 등록 및 매칭 과정 ▼
부동산 중개인이 매물을 등록하고 프랜차이즈 요청과 매칭하는 과정을 살펴봅시다.
  1. 김철수씨는 부동산 중개인으로 users 테이블에 ID 1번으로 등록되어 있어요.
  2. 그의 중개사 정보는 agent_profiles 테이블에 "행복한 부동산", 면허번호 "서울12345"로 저장되어 있어요.
  3. 김철수씨는 새로운 상가 매물을 properties 테이블에 등록해요 (강남역 1분 거리, 30평, 보증금 5천만원 등).
  4. 등록된 매물에 대해 ai_reports 분석을 요청하고, 해당 매물이 어떤 업종에 적합한지 확인해요.
  5. 분석 결과는 ai_report_location, ai_report_market, ai_report_profit 등의 테이블에 저장돼요.
  6. 김철수씨는 property_requests 테이블에서 '맛있는 치킨' 프랜차이즈가 요청한 매물 조건을 확인해요.
  7. 조건이 맞는 것 같아 chat_rooms을 통해 프랜차이즈 담당자에게 메시지를 보내요.
  8. 매물 정보와 AI 보고서를 messages 테이블에 첨부파일로 공유해요.
  9. 프랜차이즈가 관심을 보이면, 예비 점주도 찾아 contract_parties 테이블에 모든 관계자를 연결해요.
  10. 계약이 완료되면 activity_logs 테이블에 "매물 계약 완료" 활동이 기록돼요.
예시 4: 부동산 중개인의 프랜차이즈 매물 요청 대응 ▼
부동산 중개인이 프랜차이즈의 매물 요청에 대응하는 과정을 살펴봅시다.
  1. 김철수씨는 notifications 테이블을 통해 새로운 매물 요청 알림을 받았어요.
  2. property_requests 테이블에서 '맛있는 치킨' 프랜차이즈가 강남역 인근 1층 상가를 찾는다는 요청을 확인해요.
  3. request_preferences 테이블에서 상세 조건(30평 이상, 보증금 1억 이하 등)을 확인해요.
  4. 김철수씨는 properties 테이블에서 조건에 맞는 매물들을 필터링해요.
  5. 적합한 매물 3개를 찾아 ai_reports를 확인하고 치킨집에 적합한지 분석해요.
  6. ai_report_competition 테이블에서 주변 경쟁 업체 정보를 확인해요.
  7. 박지민 프랜차이즈 담당자에게 chat_rooms을 통해 연락해요.
  8. 매물 정보와 AI 분석 자료를 attachments 테이블을 통해 메시지에 첨부해요.
  9. 관심을 보이는 매물에 대해 현장 방문 일정을 잡고 activity_logs에 기록해요.
  10. 계약이 성사되면 contracts 테이블에 새 계약이 추가되고, 김철수씨는 중개 수수료를 받아요.
예시 5: 부동산 중개인의 데이터 기반 매물 관리 ▼
부동산 중개인이 데이터를 활용해 매물을 전략적으로 관리하는 과정을 살펴봅시다.
  1. 김철수씨는 자신이 등록한 모든 매물을 properties 테이블에서 확인해요 (agent_id = 1).
  2. 각 매물별 ai_reports 데이터를 분석해 어떤 매물이 가장 인기있는지 파악해요.
  3. activity_logs 테이블에서 사용자들이 어떤 매물에 관심을 보이는지 확인해요.
  4. franchises 테이블에서 최근 확장 중인 프랜차이즈 브랜드를 찾아요.
  5. 해당 브랜드에 적합한 매물을 ai_report_market 데이터를 바탕으로 선별해요.
  6. 선별된 매물을 property_requests와 매칭해 타겟 프랜차이즈에 messages로 제안해요.
  7. 지속적으로 regions 테이블의 지역 정보를 분석해 유망 지역의 매물을 우선적으로 확보해요.
  8. 성공적인 계약 패턴을 contracts 테이블에서 분석해 비슷한 조건의 매물을 더 확보해요.
  9. 모든 커뮤니케이션 기록은 messages 테이블에 저장되어 클라이언트와의 관계를 유지하는 데 활용해요.
  10. 이렇게 데이터 기반으로, 김철수씨는 더 효율적인 매물 중개가 가능해져요!
