데이터베이스란 무엇인가요? 🤔

데이터베이스는 여러분의 정보를 저장하는 디지털 창고예요. 우리 집에 있는 서랍장처럼, 각 서랍마다 특정 물건들을 정리해서 보관하는 것과 비슷합니다.

📱 여러분이 스마트폰에서 연락처를 저장할 때, 그 연락처는 사실 작은 데이터베이스에 저장되고 있어요!

🛒 온라인 쇼핑몰에서 물건을 살 때, 여러분의 주문 정보는 데이터베이스에 저장됩니다.

🏦 은행 계좌의 잔액, 입출금 내역도 모두 데이터베이스에 저장되어 있죠!

쉽게 생각하기

스마트폰 연락처부 = 작은 데이터베이스

엑셀 spreadsheet = 간단한 데이터베이스

도서관 책 관리 시스템 = 큰 데이터베이스

DB 구조 쉽게 이해하기 📋

데이터베이스는 여러 테이블로 구성되어 있어요. 테이블은 엑셀 시트처럼 행(가로줄)과 열(세로줄)로 구성됩니다.

📋 테이블 = 정보를 담는 표 (예: 사용자 테이블, 매물 테이블)

📝 열(Column) = 정보의 종류 (예: 이름, 이메일, 전화번호)

📌 행(Row) = 하나의 데이터 세트 (예: 한 명의 사용자 정보)

🔑 기본 키(Primary Key) = 각 행을 구분하는 고유 식별자 (예: 사용자 ID)

🔗 외래 키(Foreign Key) = 다른 테이블과 연결하는 키 (예: 사용자 ID 참조)

초간단 예시: 연락처 앱
id (기본 키) 이름 전화번호 그룹
1 김철수 010-1234-5678 친구
2 이영희 010-8765-4321 가족

사용자 사용자 정보 (Users) 👤

사용자 관련 테이블들은 시스템을 사용하는 모든 사람들의 정보를 저장해요. 프랜차이즈 담당자, 부동산 중개인, 예비 점주 등 각 사용자의 기본 정보와 역할별 세부 정보를 관리합니다.
user_roles 테이블 (사용자 역할) ▼
시스템에서 사용하는 사용자 역할 유형을 정의하는 테이블입니다.
id (INT)
역할 ID - 각 역할을 식별하는 고유 번호예요
name (VARCHAR(50))
역할 이름 - 프랜차이즈, 부동산 중개사, 예비 점주, 관리자 등의 역할 이름이에요

실제 예시: id가 1인 역할은 '프랜차이즈', id가 2인 역할은 '부동산 중개사'로 저장되어 있어요.

users 테이블 (모든 사용자 기본 정보) ▼
모든 사용자의 공통된 기본 정보를 저장하는 테이블입니다.
id (INT)
고유 사용자 ID - 각 사용자를 구분하는 고유 번호예요
email (VARCHAR(255))
로그인 이메일 - 로그인에 사용하는 이메일 주소예요 (중복 불가)
password_hash (VARCHAR(255))
암호화된 비밀번호 - 보안을 위해 실제 비밀번호가 아닌 암호화된 값을 저장해요
role_id (INT)
사용자 역할 ID - user_roles 테이블과 연결되어 사용자의 역할을 나타내요
is_deleted (TINYINT(1))
삭제 여부 - 계정 삭제 여부를 표시해요

실제 예시: 김철수(id: 1)님은 이메일이 kim@example.com이고, 역할(role_id)은 2번(부동산 중개사)입니다.

user_profiles 테이블 (사용자 프로필) ▼
사용자의 개인 정보와 프로필을 저장하는 테이블입니다.
id (INT)
프로필 ID - 각 프로필을 구분하는 고유 번호예요
user_id (INT)
사용자 ID - users 테이블과 연결해 어떤 사용자의 프로필인지 나타내요
name (VARCHAR(100))
실명 - 사용자의 실제 이름이에요
phone_number (VARCHAR(20))
전화번호 - 연락처예요

실제 예시: 김철수님(user_id: 1)의 프로필에는 실명 "김철수", 전화번호 "010-1234-5678", 직위 "대표 공인중개사"가 저장되어 있어요.

agents, franchises, owners 테이블 (역할별 추가 정보) ▼
각 역할별 사용자의 추가 정보를 저장하는 테이블들입니다.
agents 테이블
- id: 고유 식별자
- user_id: 사용자 ID (고유값)
- agency_name: 부동산 중개사무소 이름
- license_number: 공인중개사 등록번호 (암호화 저장)
franchises 테이블
- id: 고유 식별자
- user_id: 사용자 ID (고유값)
- company_name: 프랜차이즈 회사명
- company_number: 사업자등록번호 (암호화 저장)
- description: 회사 소개
owners 테이블
- id: 고유 식별자
- user_id: 사용자 ID (고유값)
- interests: 관심 프랜차이즈/지역 등

실제 예시:

1. 김철수님(user_id: 1)은 agents 테이블에 agency_name: "행복한 부동산", license_number: (암호화된 "서울12345") 정보가 있어요.

2. 박지민님(user_id: 3)은 franchises 테이블에 company_name: "맛있는 치킨", description: "전국 500개 매장을 운영하는 치킨 프랜차이즈" 정보가 있어요.

쉽게 이해하기

🔑 user_roles는 사용자 유형(프랜차이즈, 부동산 중개사 등)을 정의해요.

👤 users는 모든 사용자의 로그인 정보를 저장해요.

📋 user_profiles는 모든 사용자의 개인 정보를 저장해요.

🏠 agents, franchises, owners는 각 역할별 추가 정보를 저장해요.

📌 모든 역할별 테이블은 user_id로 users 테이블과 연결됩니다!

매물 매물 정보 (Properties) 🏢

매물 관련 테이블들은 부동산 매물과 프랜차이즈 매물 요청 정보를 저장해요. 위치, 면적, 가격, 파일 등 다양한 정보를 체계적으로 관리합니다.
properties 테이블 (매물 기본 정보) ▼
매물의 기본 정보를 저장하는 테이블입니다.
id (INT)
고유 매물 ID - 각 매물을 구분하는 고유 번호예요
agent_id (INT)
등록한 부동산 중개사 ID - 이 매물을 등록한 부동산 중개사를 가리켜요
title (VARCHAR(200))
매물 제목 - 매물을 소개하는 제목이에요
address (TEXT)
매물 주소 - 매물의 정확한 위치예요
area_m2 (DECIMAL(10,2))
면적 (제곱미터) - 매물의 크기를 제곱미터로 표시해요
floor (INT)
층수 - 매물이 위치한 층이에요
status (VARCHAR(20))
상태 - 현재 매물의 상태 (available, pending, sold)예요

실제 예시: '강남역 1분 거리 신축 상가'(id: 5)는 김철수 중개사(agent_id: 1)가 등록한 매물로, 주소는 '서울시 강남구 강남대로 123', 면적은 30㎡, 층수는 1층, 상태는 'available'입니다.

property_prices 테이블 (매물 가격 정보) ▼
매물의 다양한 가격 정보를 저장하는 테이블입니다.
id (INT)
고유 식별자 - 각 가격 정보를 구분하는 고유 번호예요
property_id (INT)
매물 ID - 어떤 매물의 가격인지 연결해요
field_name (VARCHAR(50))
가격 항목 - 어떤 종류의 가격인지 구분해요(보증금, 월세, 매매가 등)
value (DECIMAL(15,2))
금액 - 실제 가격 금액이에요

실제 예시: 강남역 상가(property_id: 5)의 가격 정보는 여러 행으로 저장되어 있어요:

1. field_name: "deposit", value: 50000000 (보증금 5천만원)

2. field_name: "monthly_rent", value: 2000000 (월세 200만원)

3. field_name: "maintenance_fee", value: 500000 (관리비 50만원)

property_files 테이블 (매물 파일) ▼
매물의 사진, 도면 등 파일 정보를 저장하는 테이블입니다.
id (INT)
고유 식별자 - 각 파일을 구분하는 고유 번호예요
property_id (INT)
매물 ID - 어떤 매물의 파일인지 연결해요
file_type (VARCHAR(50))
파일 유형 - 사진, 도면, 계약서 등 파일의 종류예요
file_path (VARCHAR(255))
파일 경로 - 서버에 저장된 파일의 위치예요
is_primary (TINYINT(1))
대표 이미지 여부 - 매물의 대표 이미지인지 여부예요

실제 예시: 강남역 상가(property_id: 5)에는 여러 파일이 연결되어 있어요:

1. 대표 사진(is_primary: 1): "/uploads/properties/5/main.jpg"

2. 내부 사진: "/uploads/properties/5/inside1.jpg"

3. 도면: "/uploads/properties/5/floor_plan.pdf"

property_requests 테이블 (매물 요청) ▼
프랜차이즈에서 특정 조건의 매물을 찾아달라고 요청하는 정보를 저장하는 테이블입니다.
id (INT)
요청 ID - 각 요청을 구분하는 고유 번호예요
franchise_id (INT)
요청한 프랜차이즈 ID - 이 요청을 한 프랜차이즈를 가리켜요
title (VARCHAR(200))
요청 제목 - 매물 요청의 제목이에요
description (TEXT)
요청 설명 - 요청에 대한 상세 설명이에요
region_code (VARCHAR(20))
지역 코드 - 매물을 찾고 싶은 지역 코드예요
contact_name (VARCHAR(100))
담당자 이름 - 요청을 담당하는 사람의 이름이에요
status (VARCHAR(20))
상태 - 요청의 현재 진행 상태예요

실제 예시: '맛있는 치킨' 프랜차이즈(franchise_id: 3)는 '강남 지역 신규 매장 물건 요청'이라는 제목으로, 강남 지역(region_code: '11230')에서 신규 매장 물건을 찾아달라고 요청했습니다. 담당자는 '박지민', 상태는 'pending'입니다.

property_request_conditions, property_request_budget 테이블 ▼
매물 요청의 상세 조건과 예산 정보를 저장하는 테이블입니다.
property_request_conditions 테이블
- id: 고유 식별자
- request_id: 매물 요청 ID
- property_type_id: 매물 유형 ID
- transaction_type: 거래 유형 ID
- area_min, area_max: 최소/최대 면적
- floor_min, floor_max: 최소/최대 층수
property_request_budget 테이블
- id: 고유 식별자
- request_id: 매물 요청 ID
- deposit_min, deposit_max: 최소/최대 보증금
- rent_min, rent_max: 최소/최대 임대료
- maintenance_fee_max: 최대 관리비
- right_price_max: 최대 권리금

실제 예시: '맛있는 치킨'의 요청(request_id: 5)의 조건은 다음과 같아요:

- conditions 테이블: 상가 유형(property_type_id: 1), 임대(transaction_type: 2), 최소 66㎡(area_min: 66), 1층(floor_min: 1)

- budget 테이블: 보증금 최대 1억원(deposit_max: 100000000), 월세 최대 300만원(rent_max: 3000000)

쉽게 이해하기

🏢 properties 테이블은 매물의 기본 정보(위치, 면적, 층수 등)를 담아요.

💰 property_prices는 매물의 가격 정보(보증금, 월세, 관리비 등)를 담아요.

🖼️ property_files는 매물의 사진과 파일을 담아요.

🔍 property_requests는 프랜차이즈가 찾는 매물 조건을 담아요.

🎯 conditions와 budget은 요청의 상세 조건과 예산을 담아요.

계약 계약 정보 (Contracts) 📝

계약 관련 테이블들은 부동산 거래 계약에 관한 모든 정보를 저장해요. 계약 유형, 조건, 파일, 상태 변경 내역 등을 관리합니다.
contract_types 테이블 (계약 유형) ▼
다양한 계약 유형 정보를 저장하는 테이블입니다.
id (INT)
고유 식별자 - 각 계약 유형을 구분하는 고유 번호예요
contract_type (VARCHAR(50))
계약 유형 - 임대차, 매매, 프랜차이즈 계약 등의 유형이에요
description (TEXT)
설명 - 계약 유형에 대한 설명이에요

실제 예시: id가 1인 계약 유형은 "commercial_lease"(상가 임대차 계약), id가 2인 계약 유형은 "franchise_agreement"(프랜차이즈 계약)입니다.

contracts 테이블 (계약 기본 정보) ▼
계약의 기본 정보를 저장하는 테이블입니다.
id (INT)
계약 ID - 각 계약을 구분하는 고유 번호예요
property_id (INT)
대상 매물 - 계약 대상인 매물을 연결해요
franchise_id (INT)
프랜차이즈 ID - 계약의 프랜차이즈 당사자예요
agent_id (INT)
부동산 중개사 ID - 계약을 중개한 부동산 중개사예요
contract_type_id (INT)
계약 유형 ID - 어떤 유형의 계약인지 연결해요
signed_at (TIMESTAMP)
계약 체결 시간 - 계약이 체결된 날짜와 시간이에요
status (VARCHAR(20))
상태 - 계약의 현재 상태(draft, in_progress, completed, cancelled)예요

실제 예시: 강남역 상가(property_id: 5)에 대한 계약(id: 7)은 '맛있는 치킨'(franchise_id: 3)과 김철수 부동산(agent_id: 1) 사이에 체결된 '상가 임대차 계약'(contract_type_id: 1)으로, 상태는 'completed'입니다.

contract_terms 테이블 (계약 조건) ▼
계약의 금액 관련 조건을 저장하는 테이블입니다.
id (INT)
고유 식별자 - 각 계약 조건을 구분하는 고유 번호예요
contract_id (INT)
계약 ID - 어떤 계약의 조건인지 연결해요
field_name (VARCHAR(50))
항목 이름 - 보증금, 월세 등 어떤 항목인지 구분해요
value (DECIMAL(15,2))
금액 - 실제 금액 값이에요

실제 예시: 계약 7번(contract_id: 7)의 조건들은 다음과 같아요:

1. field_name: "deposit", value: 50000000 (보증금 5천만원)

2. field_name: "monthly_rent", value: 2000000 (월세 200만원)

3. field_name: "contract_period", value: 24 (계약기간 24개월)

contract_files, contract_status_logs 테이블 ▼
계약 관련 파일과 상태 변경 내역을 저장하는 테이블입니다.
contract_files 테이블
- id: 고유 식별자
- contract_id: 계약 ID
- file_path: 파일 경로
- file_type: 파일 유형 (계약서, 첨부자료 등)
- file_name: 파일명
- uploaded_at: 업로드 시간
contract_status_logs 테이블
- id: 고유 식별자
- contract_id: 계약 ID
- status_before: 이전 상태
- status_after: 변경된 상태
- changed_by_user_id: 변경한 사용자 ID
- changed_at: 변경 시간
- note: 메모

실제 예시:

1. 계약 7번에는 "계약서.pdf", "부동산 감정서.pdf" 등의 파일이 저장되어 있어요

2. 계약 7번의 상태는 'draft'에서 'in_progress'로, 다시 'completed'로 변경되었으며 각각의 변경 내역이 logs 테이블에 기록되어 있어요

쉽게 이해하기

📄 contract_types는 계약의 유형을 정의해요.

📑 contracts는 계약의 기본 정보를 담아요.

💰 contract_terms는 계약의 금액 조건을 저장해요.

📎 contract_files는 계약 관련 문서를 관리해요.

📝 contract_status_logs는 계약 상태 변경 내역을 기록해요.

테이블 간의 관계 이해하기 🔄

데이터베이스에서 가장 중요한 것은 테이블들이 서로 어떻게 연결되어 있는지 이해하는 것이에요. 이것을 '관계'라고 합니다.

주요 관계:

1. 사용자와 역할: users 테이블의 role_id는 user_roles 테이블의 id를 참조해 어떤 역할인지 알려줘요.

2. 사용자와 프로필: user_profiles, agents, franchises, owners 테이블은 모두 users 테이블의 id를 참조해 특정 사용자를 가리켜요.

3. 매물과 등록자: properties 테이블의 agent_id는 agents 테이블의 id를 참조해 어떤 중개사가 등록했는지 알려줘요.

4. 매물과 가격: property_prices 테이블의 property_id는 properties 테이블의 id를 참조해 어떤 매물의 가격인지 알려줘요.

5. 계약과 관련자: contracts 테이블은 property_id, franchise_id, agent_id를 통해 각 테이블과 연결돼요.

핵심 관계 이해하기

이 데이터베이스의 핵심 흐름은 다음과 같아요:

1. 부동산 중개사(agents)가 매물(properties)을 등록해요.

2. 프랜차이즈(franchises)가 매물 요청(property_requests)을 등록해요.

3. 중개사와 프랜차이즈가 적합한 매물에 대해 계약(contracts)을 체결해요.

4. 계약 조건(contract_terms)과 문서(contract_files)가 저장돼요.

5. 모든 과정에서 사용자들은 알림(notifications)을 받아요.

실생활 예시로 이해하기 📝

새로운 DB 구조를 실제 사례를 통해 이해해 봅시다.
예시 1: 부동산 중개사의 매물 등록 과정 ▼
부동산 중개사가 새 매물을 등록하는 과정을 살펴봅시다.
  1. 김철수씨는 users 테이블에 id: 1, email: "kim@example.com", role_id: 2(부동산 중개사)로 등록되어 있어요.
  2. 그의 기본 정보는 user_profiles 테이블에 name: "김철수", phone_number: "010-1234-5678"로 저장되어 있어요.
  3. 부동산 중개사 정보는 agents 테이블에 agency_name: "행복한 부동산", license_number: (암호화된 "서울12345")로 저장돼요.
  4. 김철수씨가 강남역 상가 매물을 properties 테이블에 등록해요 (title: "강남역 1분 거리 신축 상가", address: "서울시 강남구 강남대로 123" 등).
  5. 매물의 가격 정보는 property_prices 테이블에 여러 행으로 저장돼요 (보증금 5천만원, 월세 200만원, 관리비 50만원 등).
  6. 매물 사진들은 property_files 테이블에 저장되고, 대표 이미지는 is_primary: 1로 표시돼요.
  7. "맛있는 치킨" 프랜차이즈가 property_requests 테이블에 등록한 요청 조건과 매칭되면, 김철수씨에게 notifications 테이블을 통해 알림이 가요.
  8. 프랜차이즈와 협의가 완료되면 contracts 테이블에 새 계약이 생성되고, contract_type_id: 1(상가 임대차 계약)으로 설정돼요.
  9. 계약 조건은 contract_terms 테이블에 저장되고, 계약서 파일은 contract_files 테이블에 저장돼요.
  10. 계약이 완료되면 status가 'completed'로 변경되고, 이 변경 내역은 contract_status_logs 테이블에 기록돼요.
예시 2: 프랜차이즈의 매물 요청 과정 ▼
프랜차이즈 회사가 특정 조건의 매물을 요청하는 과정을 살펴봅시다.
  1. 박지민씨는 users 테이블에 id: 3, email: "park@example.com", role_id: 1(프랜차이즈)로 등록되어 있어요.
  2. 그의 기본 정보는 user_profiles 테이블에, 회사 정보는 franchises 테이블에 저장되어 있어요.
  3. 박지민씨는 property_requests 테이블에 새 요청을 등록해요 (title: "강남 지역 신규 매장 물건 요청", region_code: "11230", status: "pending").
  4. 요청의 상세 조건은 property_request_conditions 테이블에 저장돼요 (property_type_id: 1(상가), area_min: 66, floor_min: 1 등).
  5. 예산 정보는 property_request_budget 테이블에 저장돼요 (deposit_max: 100000000, rent_max: 3000000 등).
  6. 요청이 등록되면 지역 내 부동산 중개사들에게 notifications 테이블을 통해 알림이 가요.
  7. 김철수 중개사가 조건에 맞는 매물을 제안하면, 박지민씨는 매물을 검토해요.
  8. 매물이 마음에 들면 contracts 테이블에 새 계약이 생성되고, 계약 과정이 시작돼요.
  9. 계약 진행 상태는 contract_status_logs 테이블에 기록되어 모든 변경 내역을 추적할 수 있어요.
예시 3: 계약 체결 과정 ▼
부동산 중개사와 프랜차이즈 사이의 계약 체결 과정을 살펴봅시다.
  1. 김철수 중개사(agent_id: 1)가 등록한 강남역 상가(property_id: 5)에 대해 박지민(franchise_id: 3)이 관심을 보여요.
  2. 두 사람이 계약 조건에 합의하면, contracts 테이블에 새 계약이 생성돼요 (id: 7, property_id: 5, franchise_id: 3, agent_id: 1, contract_type_id: 1, status: "draft").
  3. 계약의 금액 조건들은 contract_terms 테이블에 저장돼요 (보증금, 월세, 계약 기간 등).
  4. 계약서가 작성되면 contract_files 테이블에 파일이 업로드돼요.
  5. 김철수씨가 계약에 서명하면 is_signed_by_agent가 1로 변경되고, 박지민씨가 서명하면 is_signed_by_franchise가 1로 변경돼요.
  6. 모든 서명이 완료되면 status가 'completed'로 변경되고, 이 변경 내역은 contract_status_logs 테이블에 기록돼요.
  7. 계약 체결이 완료되면 양측에게 notifications 테이블을 통해 알림이 전송돼요.
  8. 매물의 상태도 'sold' 또는 'pending'으로 properties 테이블에서 업데이트 돼요.
새 DB 구조의 장점

이 새로운 DB 구조는 이전보다 더 체계적이고 유연해요!

1. 역할 기반 설계: 사용자의 역할이 명확하게 분리되어 있어요.

2. 유연한 가격 구조: 각 매물은 다양한 가격 항목을 유연하게 가질 수 있어요.

3. 계약 과정 추적: 계약의 모든 변경 사항이 기록되어 투명하게 관리돼요.

4. 파일 관리: 매물과 계약 관련 파일이 체계적으로 관리돼요.

5. 알림 시스템: 다양한 이벤트에 대한 알림을 효과적으로 전달할 수 있어요.
